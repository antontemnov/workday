## 1. Обзор <a name="overview"></a>

`ActivityEvaluator` хранит для каждой сессии **activity score** — числовую оценку
уверенности в том, что разработчик всё ещё работает. Score растёт при git-активности
(diff dynamics, коммиты) и линейно затухает каждый poll-тик.

Когда score достигает 0 → авто-пауза. Когда появляется активность на поставленной на паузу сессии → авто-возобновление.

Ключевые цели проектирования:
- **Адаптивный таймаут**: интенсивная работа → короткий таймаут (15 мин), лёгкая работа → длинный (45 мин)
- **Кросс-репозиторная осведомлённость**: в один момент времени внимание может быть только в одном репозитории
- **Независимость от единиц времени**: алгоритм работает корректно при любом значении `diffPollSeconds`

## 2. Константы и независимость от длительности тика <a name="constants"></a>

**Все константы алгоритма выражены в человекочитаемых единицах времени (минуты, секунды).**
Тиковые значения вычисляются при создании `ActivityEvaluator`.

### Исходные константы (во временных единицах)

| Константа | Значение | Единица | Описание |
|-----------|----------|---------|----------|
| `MIN_TIMEOUT_MINUTES` | 15 | мин | Минимальный таймаут авто-паузы (для интенсивной работы) |
| `MAX_TIMEOUT_MINUTES` | 45 | мин | Максимальный таймаут авто-паузы (для лёгкой работы) |
| `EMA_WINDOW_MINUTES` | 10 | мин | Окно сглаживания EMA (скорость реакции) |
| `COMMIT_BONUS_SECONDS` | 150 | сек | Дополнительные очки от коммита (в эквиваленте таймаута) |

### Производные константы (в тиках)

Все вычисляются из `diffPollSeconds` (значение из конфига, по умолчанию 30):

```
tickSeconds = config.session.diffPollSeconds

MIN_TIMEOUT_TICKS  = MIN_TIMEOUT_MINUTES * 60 / tickSeconds
MAX_TIMEOUT_TICKS  = MAX_TIMEOUT_MINUTES * 60 / tickSeconds
EMA_WINDOW_TICKS   = EMA_WINDOW_MINUTES * 60 / tickSeconds
EMA_ALPHA          = 1 / EMA_WINDOW_TICKS
COMMIT_BONUS       = COMMIT_BONUS_SECONDS / tickSeconds
```

### Таблица производных значений для разных `diffPollSeconds`

| Конфиг | MIN_TICKS | MAX_TICKS | EMA_ALPHA | COMMIT_BONUS |
|--------|-----------|-----------|-----------|--------------|
| 15с    | 60        | 180       | ~0.025    | 10           |
| 30с    | 30        | 90        | ~0.05     | 5            |
| 45с    | 20        | 60        | ~0.075    | ~3           |
| 60с    | 15        | 45        | ~0.10     | ~3           |

Диапазон таймаута [15, 45] минут сохраняется при любой длительности тика.

## 3. Модель интенсивности на базе EMA <a name="ema"></a>

**Экспоненциальное скользящее среднее (EMA)** отслеживает насколько интенсивно работает разработчик.

### Формула

```
Каждый тик:
  hasActivity = hasDynamics || hasCommit
  activityLevel = hasActivity ? 1.0 : 0.0

  intensityEMA = EMA_ALPHA × activityLevel + (1 - EMA_ALPHA) × intensityEMA
```

EMA использует **бинарный вход** (есть активность или нет). Магнитуда дельты
применяется отдельно к очкам за активность (см. §5), а не к EMA. Это сохраняет
чистую семантику EMA: она отвечает на вопрос «как часто разработчик производит
изменения?», а не «насколько большие эти изменения?».

### Поведение EMA

EMA варьируется от 0.0 (нет недавней активности) до ~1.0 (непрерывная активность).
При `EMA_ALPHA ≈ 0.05` (10-минутное окно, тики 30с):

| Паттерн работы | Установившееся EMA | Формула расчёта |
|---------------|-------------------|----------------|
| Каждый тик (интенсивная) | ~1.00 | Сходится к 1.0 |
| Каждый 3-й тик | ~0.33 | α / (1 - (1-α)^3) при α=0.05 |
| Каждый 5-й тик | ~0.22 | α / (1 - (1-α)^5) |
| Каждый 10-й тик (5 мин промежуток) | ~0.12 | α / (1 - (1-α)^10) |
| Каждый 30-й тик (15 мин промежуток) | ~0.05 | α / (1 - (1-α)^30) |

### Затухание EMA во время простоя

При 10-минутном окне EMA сохраняет память через короткие перерывы:

| Время простоя | EMA (была 0.65) | Смысл |
|--------------|----------------|-------|
| 0 мин | 0.65 | Только остановился |
| 2.5 мин | 0.53 | Ещё помнит интенсивность |
| 5 мин | 0.39 | Умеренная память |
| 10 мин | 0.23 | Начинает забывать |
| 15 мин | 0.14 | Почти забыл |
| 30 мин | 0.03 | Практически забыл |
| 45 мин | ~0 | Чистый лист |

### EMA во время паузы

- **Ручная пауза**: полная заморозка. Evaluator не получает сессию → EMA, score
  и всё состояние остаются ровно такими, какими были. При resume обработка
  продолжается с замороженного состояния. Замороженный score естественно
  выполняет роль grace period.
- **Авто-пауза** (IdleTimeout/Superseded): evaluator продолжает получать сессию.
  Score обновляется нормально (может накапливаться при активности).
  EMA продолжает обновляться каждый тик.

## 4. Адаптивный потолок score <a name="adaptive-max"></a>

Основной механизм: **потолок score адаптируется в зависимости от интенсивности работы**.

### Формула

```
dynamicMaxScore = MAX_TIMEOUT_TICKS - (MAX_TIMEOUT_TICKS - MIN_TIMEOUT_TICKS) × min(1.0, intensityEMA)
```

Константа нормализации не нужна — EMA с бинарным входом естественно достигает 1.0
при интенсивной работе и остаётся низкой при лёгкой.

### Таблица соответствий (при diffPollSeconds=30)

| EMA | dynamicMaxScore (тики) | Эквивалент таймаута |
|-----|------------------------|---------------------|
| 0.00 | 90 | 45 мин |
| 0.05 | 87 | 43.5 мин |
| 0.12 | 83 | 41.5 мин |
| 0.22 | 77 | 38.5 мин |
| 0.33 | 70 | 35 мин |
| 0.50 | 60 | 30 мин |
| 0.75 | 45 | 22.5 мин |
| 1.00 | 30 | 15 мин |

### Очки за активность

Каждое событие dynamics даёт очки, пропорциональные адаптивному потолку,
с учётом магнитуды дельты (см. §5):

```
ACTIVITY_RATIO = 0.5
magnitudeBonus = 1 + min(1, log2(1 + |addedDelta| + |removedDelta|) / MAGNITUDE_SCALE) × 0.5
activityPoints = dynamicMaxScore × ACTIVITY_RATIO × magnitudeBonus
```

`magnitudeBonus` варьируется от ×1.0 (нет изменений / только коммит) до ×1.5 (128+ строк).
Это значит, что объёмные изменения быстрее наполняют буфер score, но РАЗМЕР буфера
(dynamicMaxScore) определяется исключительно EMA (частотой активности).

| EMA | dynamicMax | activityPoints (1 строка) | activityPoints (30 строк) |
|-----|-----------|--------------------------|--------------------------|
| 1.00 | 30 | 15.5 | 20.3 |
| 0.33 | 70 | 36.2 | 47.3 |
| 0.12 | 83 | 42.9 | 56.1 |
| 0.00 | 90 | 46.5 | 60.8 |

### Обновление score за один тик

```
1. если (hasDynamics):  score += activityPoints  (с magnitudeBonus)
2. если (hasCommit):    score += COMMIT_BONUS
3. score = min(score, dynamicMaxScore)    // ограничиваем адаптивным потолком
4. score = max(0, score - BASE_DECAY)     // BASE_DECAY = 1 за тик всегда
5. если (score == 0): → решение AutoPause
```

Важно: BASE_DECAY всегда равен 1 за тик. Адаптация происходит через dynamicMaxScore
и activityPoints, а не через переменную скорость затухания. Это сохраняет модель простой.

## 5. Обогащение через магнитуду дельты <a name="magnitude"></a>

Магнитуда дельты (количество изменённых строк) влияет на **очки за активность**, а не на EMA.
EMA отслеживает частоту активности (бинарно: есть или нет).
Магнитуда отслеживает интенсивность каждого события (сколько строк изменено).

### Проектирование: разделение ответственности

| Метрика | Определяет | На какой вопрос отвечает |
|---------|-----------|-------------------------|
| EMA (бинарная) | dynamicMaxScore (размер буфера) | «Как часто разработчик производит изменения?» |
| Магнитуда | activityPoints (скорость наполнения) | «Насколько большие изменения?» |

Это разделение избавляет от необходимости в константе нормализации EMA (убранный `EMA_SATURATION`).
EMA с бинарным входом естественно достигает 1.0 при каждотиковой активности.

### Формула

```
magnitudeBonus = 1 + min(1, log2(1 + |addedDelta| + |removedDelta|) / MAGNITUDE_SCALE) × 0.5

MAGNITUDE_SCALE = 7  // log2(128) ≈ 7 → 127+ строк = максимальный бонус

activityPoints = dynamicMaxScore × ACTIVITY_RATIO × magnitudeBonus
```

### Таблица бонуса магнитуды

| Изменено строк | log2(1+n)/7 | magnitudeBonus | Эффект |
|----------------|-------------|----------------|--------|
| 0 (только коммит) | 0 | ×1.00 | Базовые очки |
| 1 | 0.14 | ×1.07 | Минимальная добавка |
| 3 | 0.29 | ×1.14 | Лёгкая правка |
| 7 | 0.43 | ×1.21 | Умеренная правка |
| 15 | 0.57 | ×1.29 | Активный кодинг |
| 30 | 0.71 | ×1.36 | Интенсивный кодинг |
| 63 | 0.86 | ×1.43 | Очень интенсивный |
| 127+ | 1.00 | ×1.50 | Максимум (ограничено) |

### Эффект

Бонус магнитуды ×1.0–×1.5 означает, что объёмные изменения наполняют буфер score до 50% быстрее.
Это влияет на скорость накопления score, а не на потолок таймаута:

- Разработчик, меняющий 127+ строк за тик, наполняет буфер на 50% быстрее
- Но РАЗМЕР буфера одинаковый (определяется только EMA)
- Практический эффект: меньше тиков нужно для достижения cap после перерыва

## 6. Кросс-репозиторное лидерство <a name="leadership"></a>

### Принцип

**В один момент времени только одна сессия может быть «лидером» (активно трекать время).**
Лидер определяется сравнением **нормализованных score** между всеми сессиями.
Никаких специальных счётчиков или порогов — существующий механизм score всё решает.

### Нормализованный score

```
normalizedScore = score / dynamicMaxScore    // 0.0 .. 1.0
```

Все сессии приводятся к единой шкале вне зависимости от адаптивного таймаута.
Score тяжёлого кодера 30/30 (1.0) и лёгкого кодера 80/90 (0.89) теперь
можно сравнивать напрямую.

### Механизм

```
На каждый вызов processAllTicks():
  1. Вычислить score для ВСЕХ сессий (кроме ручных пауз)
  2. Для каждой сессии: normalizedScore = score / dynamicMaxScore
  3. Сессия с НАИВЫСШИМ normalizedScore = лидер
  4. Все остальные сессии с score > 0: на паузу (PauseSource.Superseded)
  5. Сессии с score == 0: на паузу (PauseSource.IdleTimeout)

Важно: сессии-нелидеры НЕ замораживаются в evaluator.
  Их score продолжает обновляться (накопление при активности, затухание при простое).
  Это позволяет им конкурировать и обогнать текущего лидера.
```

### Почему нормализация?

Сырые score имеют разные шкалы: у новой сессии (dynamicMax=90) одна dynamics даёт 46 очков,
а у тяжёлого кодера (dynamicMax=30) потолок = 30.
Сырое сравнение давало бы мгновенные ложные переключения.

С нормализацией:
- Стрей dynamics в B: normB = 46/87 = 0.53 vs normA = 30/30 = 1.0 → A побеждает
- Два dynamics подряд в B: normB = 87/87 = 1.0 vs normA = 28/30 = 0.93 → B побеждает
- Возврат в A: normA = 30/30 = 1.0 vs normB = 85/87 = 0.98 → A побеждает мгновенно

### Несколько репо с активностью (один тик)

Если два репо оба имеют dynamics в одном тике:
- Лидер — тот, у кого выше нормализованный score
- При равенстве: оба остаются активными (крайне редко, обычно одна задача)

### PauseSource

```typescript
enum PauseSource {
  Manual = 'manual',                     // workday pause
  IdleTimeout = 'idle_timeout',          // score достиг 0
  Superseded = 'superseded',             // другая сессия имеет выше нормализованный score
  TeamsAway = 'teams_away',             // Фаза 3
}
```

### Ключевое отличие от прежнего AttentionSteal

| | Старое: AttentionSteal | Новое: Лидерство |
|---|---|---|
| Обнаружение | Отдельный счётчик (consecutiveActiveTicks) | Существующий score |
| Порог | Фиксированный (4 тика / 2 мин) | Органический (пересечение нормализованных score) |
| Ложные срабатывания | Rebase мог сработать | Стрей dynamics не побеждает (norm < 1.0) |
| Сложность | Дополнительное состояние per session | Нет доп. состояния |
| Ребалансировка | Жёсткий порог, бинарно | Плавная, естественная |

## 7. Жизненный цикл score и машина состояний <a name="score-lifecycle"></a>

### Машина состояний

```
                 стал лидером (score > 0)
  Pending ───────────────────────────────► Active
    │                                        │  ▲
    │                                        │  │ вернул лидерство
    │                                        │  │ (закрытие паузы)
    │                                        ▼  │
    │                                    Active(paused)
    │                                    Superseded / IdleTimeout
    │                                        │
    │  checkout / day boundary / stop         │
    └───────────────► Closed ◄───────────────┘
```

Состояния:
- **Pending**: сессия существует, evaluator считает score, время НЕ трекается.
  `startedAt` ещё не установлен. Промоция только при score > 0 И наивысшем normalizedScore.
- **Active**: сессия была или является лидером, время трекается.
  `startedAt` устанавливается в момент промоции Pending → Active.
  Может быть на паузе (Superseded/IdleTimeout), но остаётся Active.
- **Closed**: сессия завершена. `closedBy` записывает причину.

Важно: **Active никогда не возвращается в Pending.** Потеря лидерства = пауза, не понижение.

### Открытие сессии

```
score = 0
intensityEMA = 0
state = Pending
startedAt = null (устанавливается при промоции в Active)
```

Сессия начинается в состоянии `Pending`. Evaluator получает её и считает score.
Трекинг времени не начинается, пока сессия не станет лидером.

### Pending → Active (промоция)

```
Происходит когда:
  1. score > 0 (была хотя бы одна активность)
  2. normalizedScore наивысший среди всех сессий

  startedAt = now
  state = Active
  Трекинг времени начинается
```

**Оба условия обязательны.** Сессия с score = 0 не может стать лидером,
даже если она единственная. Это защищает от промоции «пустых» сессий.

Для одного репо: первый dynamics → score > 0 → единственная сессия → лидер → Active.
Для нескольких репо: сессия может оставаться Pending пока другая сессия лидирует,
накапливая score до момента обгона.

### Нормальная работа (Active сессия)

```
Каждый тик:
  1. Обновить EMA (бинарно: активность=1, простой=0)
  2. Вычислить dynamicMaxScore
  3. Начислить очки за активность (с бонусом магнитуды)
  4. Применить затухание (BASE_DECAY = 1)
  5. Сравнить нормализованные score по всем сессиям
  6. Если по-прежнему лидер → продолжить трекинг
  7. Если score == 0 → пауза IdleTimeout
  8. Если другая сессия имеет выше normalized score → пауза Superseded
```

### Авто-пауза: IdleTimeout (score == 0)

```
При IdleTimeout (Active сессия, score достиг 0):
  Запись Pause: PauseSource.IdleTimeout
  Evaluator продолжает обрабатывать (score обновляется, EMA затухает)
  Возобновление: когда приходит dynamics/commit → score > 0 → может снова стать лидером
```

### Авто-пауза: Superseded (потеря лидерства)

```
При Superseded (Active сессия, другая сессия имеет выше normalized score):
  Запись Pause: PauseSource.Superseded
  Evaluator продолжает обрабатывать (score и EMA обновляются)
  Score может быть > 0 — сессия может конкурировать и вернуть лидерство
  Возобновление: когда normalized score снова станет наивысшим → закрытие паузы
```

**Ключевое: Superseded-сессии НЕ замораживаются.** Их состояние в evaluator
продолжает обновляться. Они могут вернуть лидерство, набрав более высокий
нормализованный score.

### Авто-возобновление

```
Когда Active-сессия на паузе снова становится лидером:
  Закрываем запись Pause (устанавливаем таймстамп `to`)
  Сессия возобновляет трекинг времени

Это происходит естественно через сравнение normalized score.
Также: dynamics/commit на сессии с ручной паузой → авто-возобновление (случай «забыл resume»).
```

### Ручная пауза

```
workday pause [repo]:
  Применяется только к Active-сессиям (Pending-сессии и так не трекают время).
  SessionTracker перестаёт отправлять эту сессию в ActivityEvaluator.
  Состояние в evaluator (score, EMA) остаётся замороженным.
  Авто-возобновление ВОЗМОЖНО при git-активности (dynamics или commit).
  SessionTracker обрабатывает это: обнаруживает активность → закрывает паузу → размораживает.
```

### Ручное возобновление (workday resume)

```
workday resume:
  Закрывает ВСЕ открытые паузы (manual, idle_timeout, superseded)
  SessionTracker возобновляет отправку сессий в ActivityEvaluator
  Evaluator продолжает с замороженного состояния (score, EMA сохранены)
  Замороженный score выступает естественным grace period:
    - Интенсивный кодер (score=30): 15 мин чтобы начать кодить
    - Лёгкий кодер (score=80): 40 мин чтобы начать кодить
  Никакой специальной логики resume в evaluator не нужно
```

### Отключение авто-паузы (workday autopause off)

```
workday autopause off [repo]:
  Score продолжает обновляться (для отображения в статусе), но:
    - IdleTimeout подавляется (score=0 не вызывает паузу)
    - Смена лидерства по-прежнему работает (Superseded действует)
  Сессия остаётся активной как лидер бессрочно до:
    - workday autopause on [repo]
    - workday pause [repo]
    - Другая сессия обгоняет (Superseded)
    - Граница дня (day boundary)
    - Остановка демона
```

### Закрытие сессии

```
evaluator.removeSession(sessionId):
  Удаляем всё in-memory состояние (score, EMA)
```

## 8. Модель «стакан воды» <a name="glass-of-water"></a>

Внимание разработчика — конечный ресурс, как вода в стакане.

### Метафора

```
Обе сессии обновляют score одновременно.
Лидерство определяется тем, у кого стакан полнее (нормализованно):

Репо A (Active):   [████████░░]  norm=0.93   «лидер, трекает время»
Репо B (Pending):  [░░░░░░░░░░]  norm=0.00   «копит score, не трекает»

Разработчик переключается на B:

Тик 1:  B dynamics → B norm=0.53, A norm=0.97 → A всё ещё лидер
         B остаётся Pending (score копится, ещё не лидер)
Тик 2:  B dynamics → B norm=1.00, A norm=0.93 → B побеждает!
         B: Pending → Active (startedAt = сейчас, трекинг начинается)
         A: Active → пауза (Superseded)

Репо A (пауза):    [████████░░]  norm=0.90 (затухает, конкурирует)
Репо B (Active):   [██████████]  norm=1.00   «лидер, трекает время»

Разработчик возвращается в A:

Тик N:  A dynamics → A norm=1.00, B norm=0.95 → A побеждает!
         A: пауза закрыта (возобновляет трекинг)
         B: Active → пауза (Superseded)
```

### Почему не явный перенос?

Мы рассматривали: «репо B получает → репо A теряет (бюджет с нулевой суммой)».

Модель лидерства проще: оба стакана существуют независимо,
более полный (нормализованно) получает право трекать время.
Никакого переноса, никакого перераспределения.
Естественная конкуренция через накопление и затухание score.

## 9. CLI-команды <a name="cli-commands"></a>

### workday resume

Закрывает ВСЕ открытые паузы вне зависимости от источника (manual, idle_timeout, superseded).
Если пользователь явно написал «resume» — он знает что делает.

В будущем: интеграция Teams будет вызывать HTTP-endpoint resume для сигнала «пользователь вернулся».

### workday autopause off [repo]

Отключает IdleTimeout для указанного репозитория или для всех репо, если аргумент не указан.
Смена лидерства (Superseded) по-прежнему работает — нельзя быть лидером двух репо одновременно.

Сессия не получает IdleTimeout при score=0. Score продолжает обновляться (виден в `workday status`).

Сбрасывается при:
- `workday autopause on [repo]`
- Граница дня (чистое состояние на новый день)
- Остановка/перезапуск демона

`workday status` должен показывать предупреждение: «⚠ autopause disabled для atlas-frontend (с 14:30)»

### workday pause [repo]

Ручная пауза. Перекрывает авто-паузу. Может быть снята:
- `workday resume` — явное ручное возобновление
- Git-активность (dynamics или commit) — автоматически, на случай «забыл нажать resume»

## 10. Разбор сценариев <a name="scenarios"></a>

Все примеры используют `diffPollSeconds=30` (1 тик = 30 секунд).

### Сценарий A: Интенсивная работа, потом обед

```
Разработчик пишет код с dynamics практически каждый тик в течение 2 часов.

Фаза 1: Работа (тики 0-240, 2 часа)
  EMA → ~1.0 (бинарный вход, активность каждый тик)
  dynamicMaxScore = 30 тиков (15 мин)
  activityPoints = 15 × magnitudeBonus (~1.3 для типичных правок) ≈ 19.5
  Score: на потолке 30

Фаза 2: Начало обеденного перерыва
  Тик 241: нет dynamics, score = 30 - 1 = 29
  Тик 242: нет dynamics, score = 28
  ...
  Тик 270: нет dynamics, score = 1
  Тик 271: score = 0 → AutoPause (IdleTimeout)

  Время от последней активности до паузы: 30 тиков × 30с = 15 мин ✓

Фаза 3: Возврат с обеда (через 1 час)
  EMA во время паузы: 1.0 × (1-0.05)^120 ≈ 0.002 (полностью затухла)
  Тик 391: dynamics → AutoResume
  dynamicMaxScore ≈ 90 (EMA≈0, свежий старт с щедрым таймаутом)
  score ≈ 46.5 (activityPoints = 90 × 0.5 × magnitudeBonus)

  В лог записана пауза: с 15 мин после последней активности до возобновления ≈ 1 час паузы
```

### Сценарий B: Лёгкий кодинг (1 строка каждые 15 мин)

```
Разработчик читает код и делает редкие мелкие правки.
Промежуток между dynamics: 30 тиков (15 мин).

Установившийся режим:
  EMA → ~0.05 (очень низкая, бинарный вход каждый 30-й тик)
  dynamicMaxScore = 87 тиков (43.5 мин)
  activityPoints = 87 × 0.5 × 1.07 (магнитуда 1 строка) ≈ 46.5

Траектория score:
  Тик 0:  dynamics → score = 46.5
  Тики 1-29: затухание → score = 46.5 - 29 = 17.5
  Тик 30: dynamics → score = 17.5 + 46.5 = 64.0, ограничение 87 → 64.0
  Тики 31-59: затухание → score = 64.0 - 29 = 35.0
  Тик 60: dynamics → score = 35.0 + 46.5 = 81.5
  Тики 61-89: затухание → score = 81.5 - 29 = 52.5
  Тик 90: dynamics → score = 52.5 + 46.5 = 99.0, ограничение 87 → 87.0
  ...сходится к dynamicMaxScore (87)

  Score никогда не достигает 0 во время нормальной работы ✓

После последнего dynamics (прекращение работы):
  Score ≈ 80-87, затухание по 1/тик
  Таймаут: 80-87 тиков = 40-43 мин ✓ (в диапазоне [15,45])
```

### Сценарий C: Переключение между репозиториями

```
Два репозитория: atlas-frontend (A), appone-backend (B).

12:00  Работаю в A. Score_A=30, EMA_A≈1.0, dynamicMax_A=30.
12:05  Открыл B, начал кодить.

12:05:00  Тик 1: B dynamics. B score=46.5, dynamicMax=87, norm=0.53.
          A: нет активности. score=29, norm=0.97.
          A(0.97) > B(0.53) → A всё ещё лидер ✓

12:05:30  Тик 2: B dynamics. B score=87(cap), norm=1.00.
          A: score=28, norm=0.93.
          B(1.00) > A(0.93) → B берёт лидерство!
          A: Pause { from: "12:05:30", to: null, source: "superseded" }
          A score продолжает затухать (НЕ заморожен, evaluator всё ещё видит)

12:05:30 - 12:30  B — лидер. A на паузе, но score обновляется.

12:30  Возврат в A. Dynamics в A.
       A score затухал: ≈0 (50 тиков затухания). dynamics → +activityPoints.
       A: EMA затухла за время паузы, dynamicMax ≈ 80. activityPoints ≈ 42.
       A score = 42, norm = 42/80 = 0.53.
       B: нет dynamics. score ≈ 40 (затухает от 87), norm = 40/87 = 0.46.
       A(0.53) > B(0.46) → A возвращает лидерство!
       A пауза закрыта: { from: "12:05:30", to: "12:30:00" }

12:30+  A снова лидер. B затухает → IdleTimeout со временем.

Результат:
  A в логе: 12:00-12:05:30 активно, 12:05:30-12:30 пауза(superseded), 12:30+ активно
  B в логе: startedAt=12:05:30 (промоция из Pending), 12:05:30-12:30 активно, 12:30+ пауза
  Пересечение: 0 сек (B был Pending до момента лидерства, A на паузе с того же момента)
```

### Сценарий D: Случайное касание в неактивном репо

```
Разработчик активно работает в repo A (лидер, score=30, norm=1.0).
Случайно сохранил файл в repo B.

Тик 0: dynamics в B (1 строка)
  B: score = 46.5, dynamicMax = 87, norm = 0.53
  A: score = 29, norm = 0.97
  A(0.97) > B(0.53) → A по-прежнему лидер
  B остаётся Pending (не промотирована, т.к. A имеет выше normalizedScore)

Больше активности в B нет:
  B score затухает: 46.5 → 45.5 → ... → 0 через 47 тиков
  B закрывается как Pending при score=0 или при checkout

Результат: случайный файл не украл лидерство у A.
Сессия B осталась Pending → исключается из отчёта.
```

### Сценарий E: Коммит, потом продолжение работы

```
Разработчик кодит, коммитит, продолжает.

Тик N:   dynamics (последняя правка перед коммитом)
  Score: ~30 (интенсивный кодер, EMA≈1.0)

Тик N+1: git add . && git commit
  git diff --numstat: 0/0 (чистое рабочее дерево)
  delta: addedDelta = 0-50 = -50 → hasDynamics = TRUE
  reflog: новый коммит → hasCommit = TRUE

  Score: 30 + ~20 (activityPoints с магнитудой) + 5 (COMMIT_BONUS) = ~55, ограничение 30 → 30
  (По-прежнему на потолке, никакого штрафа от коммита)

Тик N+2: снова начал кодить → dynamics
  diff: +3/+1, delta: +3/+1 → hasDynamics = TRUE
  Score: 30 + 15 = 45, ограничение 30 → 30

Тик N+2 (альтернатива: не начал кодить):
  diff: 0/0, delta: 0/0 → hasDynamics = FALSE
  Score: 30 - 1 = 29
  Нормальное затухание начинается

Вывод: коммит сам по себе — сигнал активности (двойной буст). Score не падает.
Переход бесшовный вне зависимости от того, продолжает ли разработчик или останавливается.
```

### Сценарий F: Отключённая авто-пауза (чтение/размышления)

```
Разработчик вернулся с обеда, хочет почитать код какое-то время.

13:00  workday resume → все паузы закрыты
13:00  workday autopause off → авто-пауза отключена для всех сессий
       Статус показывает: «⚠ autopause disabled (с 13:00)»

13:00-14:30  Разработчик читает код, нет git-изменений.
       Score: 0 (нет активности). IdleTimeout подавлен (autopause off).
       Сессия остаётся Active (единственная, лидер по умолчанию). Время логируется.

14:30  Разработчик начинает кодить → появляются dynamics
       Score: 0 + activityPoints
       workday autopause on → возвращаем нормальное поведение
```

### Сценарий G: Очень медленный кодер с интервалом опроса 60с

```
diffPollSeconds = 60. Разработчик вносит изменения каждые 15 мин.

Производные константы:
  MIN_TIMEOUT_TICKS = 15 * 60 / 60 = 15 тиков
  MAX_TIMEOUT_TICKS = 45 * 60 / 60 = 45 тиков
  EMA_ALPHA = 1 / (10 * 60 / 60) = 1/10 = 0.10

Промежуток между dynamics: 15 тиков (15 мин при 60с/тик).

Установившееся EMA: α / (1 - (1-α)^15) = 0.10 / (1 - 0.90^15) ≈ 0.13
dynamicMaxScore = 45 - 30 × 0.13 = 41.1 тиков
activityPoints = 41.1 × 0.5 × 1.07 (мелкая правка) ≈ 22.0

Score после dynamics: +22.0
После 15 пустых тиков: 22.0 - 15 = 7.0 (выживает ✓)

После остановки (финальная dynamics):
  Score ≈ 35-41 → таймаут = 35-41 тиков × 60с = 35-41 мин ✓
  В диапазоне [15, 45] ✓
```

## 11. Граничные случаи <a name="edge-cases"></a>

### Восстановление после краша

При перезапуске демона `ActivityEvaluator` не имеет in-memory состояния.
Все сессии начинают с score=0, EMA=0. Существующие открытые сессии:
- Если следующий poll имеет dynamics → обрабатывается как холодный старт (щедрый таймаут)
- Если нет dynamics → score=0 → AutoPause на первом тике

Это допустимо: краш означает потерю состояния. Быстрая авто-пауза
с последующим авто-возобновлением при активности — самое безопасное поведение.

### Граница дня

Все сессии закрываются через `ClosedBy.DayBoundary`. Состояние evaluator обнуляется.
Новый день начинается полностью с чистого листа.

### Быстрое переключение между репо (пинг-понг)

Разработчик чередует репо A и B каждую минуту (фронтенд + бэкенд одной задачи).
Лидерство переходит по normalized score: кто кодит в данный момент, тот лидер.
Паузы Superseded создаются и закрываются при каждом переключении.

Если задача одна (ATL-123 в обоих репо) — пересечения не влияют на итоговый отчёт,
т.к. отчёт группирует время по задаче, а не по репозиторию.

### Три репозитория

Лидером всегда остаётся одна сессия с наивысшим normalized score:

A: Active → Superseded (B обогнал) → остаётся Active(paused)
B: Pending → Active (стал лидером) → Superseded (C обогнал)
C: Pending → Pending → Active (стал лидером)

### Dynamics от rebase/merge

Rebase может генерировать большие diff dynamics (100+ строк), которые не являются реальной разработкой.
Сейчас обрабатывается существующей классификацией (тип reflog = 'other').

Будущее улучшение: детектировать директорию `.git/rebase-merge/` и подавлять dynamics
во время rebase. Не часть текущего алгоритма.

### Сессия в состоянии Pending

Pending-сессии накапливают score в evaluator, но не трекают время.
Они конкурируют за лидерство через сравнение нормализованных score.

Pending-сессия переходит в Active когда становится лидером.
До этого она остаётся Pending вне зависимости от количества dynamics.

Pending-сессии закрываются при:
- Checkout на другую ветку → ClosedBy.CheckoutOtherTask
- Граница дня → ClosedBy.DayBoundary
- Никогда не стала Active → исключается из отчёта (время не трекалось)

## 12. Сводная таблица констант <a name="constants-summary"></a>

### Конфигурируемые (в config.json → session)

| Поле | По умолчанию | Описание |
|------|-------------|----------|
| `diffPollSeconds` | 30 | Интервал опроса. Алгоритм адаптируется к любому значению. |
| `idleTimeoutMinutes` | 20 | НЕ используется адаптивным алгоритмом напрямую. Оставлен для возможного simple-mode в будущем. |

### Константы алгоритма (захардкожены в ActivityEvaluator)

| Константа | Значение | Обоснование |
|-----------|----------|-------------|
| `MIN_TIMEOUT_MINUTES` | 15 | Минимальное практичное время обнаружения рабочей сессии |
| `MAX_TIMEOUT_MINUTES` | 45 | Максимальное терпение для очень медленного, но стабильного кодинга |
| `EMA_WINDOW_MINUTES` | 10 | Память ~15 мин, забывает после ~30 мин простоя |
| `ACTIVITY_RATIO` | 0.5 | Одна dynamics = 50% от окна таймаута (до бонуса магнитуды) |
| `MAGNITUDE_SCALE` | 7 | log2(128): 127+ строк/тик = максимальный бонус магнитуды (×1.5) |
| `MAGNITUDE_BONUS_MAX` | 0.5 | Макс. дополнительный множитель для крупных изменений (activityPoints × 1.0–1.5) |
| `COMMIT_BONUS_SECONDS` | 150 | Коммит добавляет ~2.5 мин буфера |
| `BASE_DECAY` | 1 | Всегда 1 за тик (адаптация через maxScore, а не через скорость затухания) |

### Контракт (TypeScript)

```typescript
interface ActivitySignals {
  readonly hasDynamics: boolean;
  readonly hasCommit: boolean;
  readonly deltaMagnitude: number;  // |addedDelta| + |removedDelta|
}

interface TickInput {
  readonly sessionId: string;
  readonly signals: ActivitySignals;
  readonly autoPauseDisabled: boolean;
  // Примечание: сессии на ручной паузе НЕ отправляются в evaluator (полная заморозка)
}

interface EvaluatorResult {
  readonly scores: Map<string, SessionScore>;
  readonly leaderId: string | null;  // сессия с наивысшим normalized score, null если все score=0
}

interface SessionScore {
  readonly score: number;
  readonly maxScore: number;
  readonly normalizedScore: number;  // score / maxScore (0..1)
  readonly ema: number;
  readonly isIdleTimeout: boolean;   // score == 0
}

class ActivityEvaluator {
  constructor(diffPollSeconds: number);
  processAllTicks(ticks: readonly TickInput[]): EvaluatorResult;
  removeSession(sessionId: string): void;
}
```
